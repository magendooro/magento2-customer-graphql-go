## Project: magento2-customer-graphql-go

High-performance Go drop-in replacement for Magento 2's customer-related GraphQL queries and mutations using gqlgen. Produces identical responses to Magento PHP — 72 tests including 11 field-by-field comparisons verify this.

## Architecture

- **Schema-first GraphQL** via gqlgen — edit `graph/schema.graphqls`, then `GOTOOLCHAIN=auto go run github.com/99designs/gqlgen generate`
- **Never edit** `graph/generated.go` or `graph/model/models_gen.go` — auto-generated by gqlgen
- **Magento Enterprise Edition** — customer entity uses `entity_id` (NOT `row_id` — that's catalog only)
- **Read AND write** — queries (customer, orders, isEmailAvailable) and mutations (token gen, CRUD, password, addresses)
- **JWT authentication** — Magento-compatible HS256 JWTs signed with `crypt/key`. Cross-compatible: Go tokens work with Magento PHP and vice versa. Requires `MAGENTO_CRYPT_KEY` env var.
- **Patched gqlgen** — uses fork (`florinel-chis/gqlgen`) via `replace` in go.mod for resolver error locations. Remove when upstream [#4090](https://github.com/99designs/gqlgen/pull/4090) merges.

## Project Structure

```

  schema.graphqls     Full GraphQL schema (queries, mutations, types)

  jwt/                Magento-compatible JWT creation/validation
  middleware/         Auth (JWT+oauth), CORS, cache, logging, recovery, store
  repository/         Data access (one file per domain):
    customer.go       Customer CRUD, password hashing (Argon2id/SHA256/bcrypt)
    address.go        Address CRUD
    token.go          JWT token creation, revocation via jwt_auth_revoked
    order.go          Order history (12 batch-loading methods across 10+ sales tables)
    eav_attribute.go  Custom EAV attribute loading
    group.go          Customer group lookup
    newsletter.go     Newsletter subscription
    store.go          Store/website resolution
  service/            Business logic:
    customer.go       Customer operations, lockout, password validation, mapping
    orders.go         Order service with parallel batch loading, timezone handling
tests/                Integration + comparison tests (HTTP-based via httptest)
```

## Build & Test

```bash
GOTOOLCHAIN=auto go build -o server ./cmd/server/    # build
GOTOOLCHAIN=auto go vet ./...                         # lint
GOTOOLCHAIN=auto go run github.com/99designs/gqlgen generate  # regenerate after schema changes
GOTOOLCHAIN=auto go test ./... -count=1 -timeout 120s # all 72 tests

# single test
GOTOOLCHAIN=auto go test ./tests/ -run TestCompare_CustomerFields -v -timeout 60s -count=1

# run server (port 8082 — Magento is on 8080)
MAGENTO_CRYPT_KEY="<from env.php>" DB_USER=fch DB_NAME=magento248 GOTOOLCHAIN=auto go run ./cmd/server/
```

## Key Conventions

- **Go 1.25** (via toolchain directive) — always prefix commands with `GOTOOLCHAIN=auto`
- **Error handling**: `fmt.Errorf("context: %w", err)`, use `errors.Is`/`errors.As`
- **Error messages**: match Magento exactly — capitalized, with period ("The current customer isn't authorized.")
- **Naming**: `CamelCase` exported, `camelCase` unexported, no stutter
- **Logging**: zerolog structured JSON — `log.Info().Str("key", val).Msg("message")`
- **Context**: always first parameter `ctx context.Context`
- **Auth**: `middleware.GetCustomerID(ctx)` returns customer ID (0 if unauthenticated)
- **Store**: `middleware.GetStoreID(ctx)` returns store ID for multi-store scoping
- **Password hashing**: Magento format `hash:salt:version` — 1=SHA256, 2=bcrypt, 3=Argon2id (default)
- **SQL safety**: whitelist column names, sort directions, and EAV backend types — never interpolate user input

## Magento Database Tables

**Customer:**
- `customer_entity` — main table (flat, most fields are columns)
- `customer_entity_varchar/int/text/decimal/datetime` — EAV values (only for custom attributes)
- `customer_address_entity` — addresses (flat, street is newline-separated)
- `customer_eav_attribute` — attribute visibility config
- `customer_group` — group names (General, Wholesale, Retailer)

**Authentication & Security:**
- `jwt_auth_revoked` — JWT revocation (user_type_id, user_id, revoke_before)
- `oauth_token` — legacy opaque tokens (fallback only)

**Orders:**
- `sales_order` — orders with status, totals, shipping
- `sales_order_item` — line items
- `sales_order_address` — billing/shipping addresses
- `sales_order_payment` — payment methods
- `sales_order_status` — status code → label mapping
- `sales_order_status_history` — visible comments
- `sales_invoice`, `sales_invoice_item` — invoices
- `sales_shipment`, `sales_shipment_item`, `sales_shipment_track` — shipments
- `sales_creditmemo`, `sales_creditmemo_item` — credit memos

**Config:**
- `core_config_data` — store timezone, password rules, email availability check
- `newsletter_subscriber` — newsletter status
- `store` — store/website resolution
- `directory_country_region` — region code/name lookup
- `eav_attribute`, `eav_attribute_option`, `eav_attribute_option_value` — EAV metadata

## Common Patterns

### Adding a customer field
1. Add to `Customer` type in `graph/schema.graphqls` → regenerate
2. If flat column: add to `CustomerData` struct + `GetByID()`/`GetByEmail()` SQL + `mapCustomer()`
3. If EAV: use `EAVAttributeRepository` (auto-loaded via `custom_attributes` field)

### Adding a mutation
1. Add to `Mutation` type in schema → regenerate
2. Implement in `internal/service/customer.go`
3. Wire resolver in `graph/schema.resolvers.go`
4. For authenticated mutations: check `middleware.GetCustomerID(ctx)` == 0

### Adding a field-level resolver
1. Add to `gqlgen.yml` under `models.<TypeName>.fields.<field_name>.resolver: true`
2. Regenerate → implement the resolver stub on the generated `<typeName>Resolver` struct

## Development Workflow — Lessons Learned

These rules come from real bugs caught during development. Follow them.

### Before writing SQL
- **`DESCRIBE` every table** before writing queries. Don't assume columns exist.
- **Use `COALESCE(col, 0)`** for nullable numeric columns. Scanning NULL into `float64` panics.
- **Whitelist all SQL identifiers** — sort directions, column names, table name suffixes. Never interpolate user input into SQL, even if "the GraphQL schema constrains it."
- **When adding columns to SELECT**, update **ALL** methods that query the same table (`GetByID` AND `GetByEmail`). Column count must match Scan parameter count exactly.

### Before implementing any query/mutation
- **Check Magento's actual PHP code** for the resolver. Don't assume behavior.
- **Check `core_config_data`** for config that affects behavior (e.g., `isEmailAvailable` anti-enumeration, password rules, timezone).
- **Check if Magento has a lookup table** for display values (e.g., `sales_order_status` for status labels, `eav_attribute_option_value` for option labels).

### After implementing
1. `GOTOOLCHAIN=auto go build ./... && go vet ./...`
2. `GOTOOLCHAIN=auto go test ./... -count=1 -timeout 120s` — all 72+ tests must pass
3. **Live comparison**: start the Go server, send the same query to both `:8080` (Magento) and `:8082` (Go), diff the JSON. This is the ONLY way to verify true compatibility. Testing against DB ground truth is NOT sufficient.

### Crypto/auth rules
- **Always test round-trips** for password hashing (hash→verify) and JWT (create→validate).
- **Never log tokens or secrets** — not even at DEBUG level.
- **Test JWT cross-compatibility** — Go token must auth with Magento PHP and vice versa.

### gqlgen rules
- **`resolver.go`** must ONLY contain `Resolver` struct + `NewResolver()`. Nothing else.
- **Helper functions** go in `helpers.go`, never in `schema.resolvers.go` (gets overwritten by gqlgen).
- **Never manually write gqlgen execution code** — always regenerate. Delete any hand-written `*_exec.go` files.
- **Using a gqlgen fork**: replace directive in go.mod. Remove when upstream merges. Currently patching: error response `locations` field ([#4090](https://github.com/99designs/gqlgen/pull/4090)).

### Timezone handling
- MySQL `TIMESTAMP` columns are stored in UTC internally but converted to session timezone on read.
- DSN must include `time_zone=%27%2B00%3A00%27` to get UTC from MySQL, PLUS `loc=UTC` for Go's interpretation.
- Magento converts order dates to the store timezone from `general/locale/timezone` in `core_config_data`.
- Use `OrderService.formatOrderDate()` for order timestamps, `formatDateTime()` for customer timestamps.

### Working with sandbox/cowork-generated code
- **DESCRIBE every table** referenced in SQL — sandbox may assume columns that don't exist.
- **Delete any `*_exec.go` files** and regenerate with gqlgen — sandbox can't run gqlgen and hand-writes these.
- **Clean up `.git/*.lock` files** left by crashed processes.
- **Check for nullable columns** — sandbox often uses `float64` where `COALESCE` is needed.
